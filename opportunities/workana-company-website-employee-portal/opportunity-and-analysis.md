# Workana Opportunity: Company Website Rebuild with Secure Employee Portal

## Original opportunity

We are seeking an experienced web development team or freelancer to completely recreate our existing company website. The new website must incorporate our updated brand identity, ensuring a fresh and modern look while retaining all current content and functionalities. A key component of this project is the development of a new, secure employee portal. This portal will require user authentication and specific features to facilitate internal communication and resource sharing. The ideal candidate will have strong front-end and back-end development skills, with a keen eye for detail in implementing design specifications and ensuring a seamless user experience. The site needs to be in English and Spanish and friendly for us to make changes.

## Opportunity assessment

The main challenge is not only rebuilding a website with a new look. The real engineering problem is separating two systems with different concerns:

- a public multilingual company website that reflects the new brand and remains easy to maintain;
- a secure employee portal with authentication and internal communication or resource-sharing features.

This distinction matters because the architecture, editing workflow, security model, content strategy, and long-term maintenance requirements are different.

The main risks are:

- unclear boundary between public content and internal authenticated features;
- underestimating the complexity of authentication, roles, password recovery, session handling, and secure access;
- migrating all existing content and functions without regressions;
- choosing a CMS or editing model that is not friendly for bilingual maintenance;
- mixing marketing-site needs with internal portal workflows inside one unmanaged scope;
- unclear hosting, deployment, and ownership expectations after launch.

The proposal should avoid framing this as a simple website redesign. A safer first step is to clarify the information architecture, portal scope, content migration approach, editing workflow, security model, and release boundary before committing to a larger implementation cycle.

## Recommended commercial framing

The safest commercial path is to sell a one-week Discovery Sprint first, while making clear that implementation can start immediately afterward in a fixed delivery cycle once the public-site scope, portal scope, content workflow, and security requirements are confirmed.

Suggested structure:

- Phase 1: Discovery Sprint, 1 week.
- Phase 2: architecture recommendation covering public website, employee portal, CMS/editing workflow, bilingual content strategy, editor permissions, and security model.
- Phase 3: first implementation cycle covering the agreed first operational website and portal release.

Suggested estimate:

- Immediate next step: USD 500 Discovery Sprint.
- Likely first implementation package after discovery: USD 4,000 Product Acceleration for a well-bounded public site plus moderate employee portal, or USD 6,000 Scale \& Intelligence if the portal requires stronger access control, auditability, integrations, or broader internal workflow support.
- Estimated timeline for discovery plus first implementation cycle: 5 weeks for the agreed first operational release, not necessarily every possible future portal workflow.
- Additional cycles may be required if the portal later expands into document workflows, announcements, permissions matrices, integrations, or richer internal collaboration features.

## Key questions before final commitment

- What content and functionality from the current website must be preserved exactly?
- Is there already a new brand guide, design system, or page mockups?
- What features must exist in the first version of the employee portal?
- Will the portal need roles and permissions, or only basic authenticated access?
- How should internal communication work: announcements, documents, messaging, resource directories, or another model?
- What is the preferred editing workflow for English and Spanish content, and who should be able to edit, review, and publish changes?
- Should the portal and the public site use the same CMS/admin system or separate tools?
- Is there an existing user directory or identity provider for employee authentication, or should the first version include standalone login and password recovery?
- What hosting, deployment, and ownership model do you expect after launch?
- Is there already an approved budget range and target launch date for the first operational version?
