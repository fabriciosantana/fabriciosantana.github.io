# Qualification

## Decision

`pursue discovery only`

## Summary

This is a strong fit with Fabricio Santana's positioning because it combines backend development, Django architecture, PostgreSQL integration, authentication, REST APIs, responsive web delivery, and maintainable engineering practices. The safest commercial move is to sell the Discovery Sprint first because the request defines the technology stack and core capabilities, but it does not define the business domain, data model, user roles, deployment model, frontend complexity, API scope, or first operational release boundary.

## Qualification assessment

- Budget signal or likely ability to pay: moderate. The requested stack and quality expectations suggest a real application build, not a trivial script, but no budget range is stated.
- Clarity of scope: moderate. The core stack and baseline features are clear, but the business workflow, entity model, role structure, hosting model, and exact responsive frontend expectations are not.
- Fit with positioning: strong. This aligns well with APIs, internal tools, product engineering, secure backend delivery, testing, deployment, and maintainable architecture.
- Urgency and business pain: likely real. The request points to a concrete need for a production-capable web application rather than exploratory work.
- Access to decision maker: unknown.
- Risk of uncontrolled scope: medium to high. Generic “full web app with auth, CRUD, APIs, responsive design” requests often expand into admin workflows, file uploads, permissions, audit history, notifications, reporting, search, and deployment support.
- Seriousness of client signals: good. The stack is specific, the quality bar is explicit, and the deliverables are framed around robustness, scalability, security, and maintainability.
- Likelihood of competing only on low price: medium. Django CRUD opportunities attract low-cost offers, so the proposal should differentiate through architecture quality, scope control, security, and operational readiness.

## Recommended commercial framing

Recommend selling the Discovery Sprint as the immediate next step, with a clear path to implementation right after it. Use discovery to confirm:

- the business domain and first operational release boundary;
- the core entities, relationships, and CRUD scope;
- the authentication and authorization model;
- the API surface and consumer expectations;
- the responsive frontend approach and UI complexity;
- the deployment, environment, and ownership model;
- the test, documentation, and acceptance criteria needed for launch.

The likely package after discovery is:

- `Product Acceleration` if the first version is a bounded production-ready Django application with authentication, PostgreSQL, REST APIs, responsive UI, testing, and deployment support.
- `Scale & Intelligence` if the scope expands into multi-role workflows, complex permissions, auditability, heavier integrations, broader API surface, or stronger observability and release requirements.

## Main sales angle

The proposal should make clear that the main challenge is not only creating Django models and CRUD screens. The real work is defining a secure and maintainable application architecture where authentication, data model, API contracts, responsive UI flows, validation rules, database design, testing, and deployment readiness work together as one controlled system.

The business value should be explicit:

- faster delivery of a production-capable first version;
- cleaner long-term maintainability;
- lower rework risk from unclear architecture;
- safer authentication and data handling;
- a clearer path for future expansion.
